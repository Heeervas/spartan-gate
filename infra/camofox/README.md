# Camofox

Camofox is an optional local browser backend for Hermes browser tools. Spartan
Gate runs it as a separate Compose service named `camofox` and keeps
Browserless as the default Chromium/CDP backend.

Use Camofox for anti-detection experiments and browser-tool tasks where a
Camoufox/Firefox-based browser is useful. Camofox also backs Google Meet
`mode=transcribe` when `CAMOFOX_URL` is set. Keep Browserless for CDP behavior,
Google Meet realtime audio, Browserless profile seeding, Chrome DevTools MCP,
and the persistent Browserless `main` profile broker.

In Camofox mode, `sg-hermes-meet setup` and `sg-hermes-meet-join` use
`CAMOFOX_USER_ID` and ignore stale `HERMES_MEET_CDP_URL` values.

## Enable

Camofox starts as an internal service, but Hermes will not use it unless
`CAMOFOX_URL` is set. `CAMOFOX_URL` is the canonical selector; when
`COMPOSE_PROFILES` is set it must match:

```env
COMPOSE_PROFILES=camofox
CAMOFOX_URL=http://camofox:9377
CAMOFOX_ACCESS_KEY=<private-key>
```

`CAMOFOX_ACCESS_KEY` is passed to the Camofox service and to Hermes. Camofox
protects most routes with `Authorization: Bearer <key>` when this key is set.
`GET /health` intentionally stays unauthenticated for Docker healthchecks, so a
successful healthcheck alone does not prove browser/auth integration.

## Runtime State

Camofox state is separate from Browserless state.

- Public default: `runtime/camofox/`
- Private override: `SPARTAN_CAMOFOX_DATA_PATH`
- Container path: `/root/.camofox`

Camofox persistence stores Playwright `storageState` under
`CAMOFOX_PROFILE_DIR`, keyed by Camofox/Hermes user IDs. It is not a Chromium
`--user-data-dir` and is not compatible with `runtime/browserless/profiles`.

For manual login profiles, use stable IDs generated by the helpers:

```text
main -> spartan-camofox-main
alt  -> spartan-camofox-alt
```

`sg-camofox-profile-save` only verifies/checkpoints state. It does not delete a
Camofox session, because deletion can remove persisted browser state.

To reset one helper-managed Camofox profile:

```sh
source scripts/aliases.sh
sg-camofox-profile-reset main
```

For a full manual wipe, stop `hermes` and `camofox`, then remove the directory
configured by `SPARTAN_CAMOFOX_DATA_PATH` or the public `runtime/camofox/`
directory.

## Egress

Camofox browser traffic must go through Spartan Gate's outbound policy. The
Compose service sets both generic proxy variables and Camofox's browser proxy
variables:

```env
PROXY_HOST=proxy
PROXY_PORT=8888
HTTP_PROXY=http://proxy:8888
HTTPS_PROXY=http://proxy:8888
```

If a target page fails unexpectedly, verify Tinyproxy logs before assuming the
browser is using the expected network path.

## Image

Spartan Gate builds a thin local wrapper image:

```env
CAMOFOX_BASE_IMAGE=ghcr.io/jo-inc/camofox-browser:1.11.2
CAMOFOX_IMAGE=spartan-gate-camofox:1.11.2
```

The wrapper does not fork Camofox. It pre-fetches Camofox's default uBlock
Origin add-on at build time. Upstream Camofox otherwise downloads that add-on
on first browser launch with Node `fetch`, which does not follow Spartan Gate's
runtime Tinyproxy path and fails while the service is correctly isolated on
`spartan_internal`.

The wrapper also makes upstream Camofox geo-IP proxy probing opt-in through
`CAMOFOX_GEOIP`. The default is `false` because Spartan Gate's normal Camofox
proxy is local Tinyproxy, not a residential proxy with a useful geolocation
target.

## Add-ons

Camofox can load extracted Firefox add-ons through `CAMOFOX_ADDONS`. The value
is a comma-separated list of container paths. For private add-ons, keep the
files on the host and bind them into the container read-only.

The private Obsidian Web Clipper install uses this mapping:

```text
Host:      SPARTAN_CAMOFOX_DATA_PATH/addons/web-clipper-obsidian
Container: /opt/camofox-addons/web-clipper-obsidian
Env:       CAMOFOX_ADDONS=/opt/camofox-addons/web-clipper-obsidian
```

Set `SPARTAN_CAMOFOX_ADDONS_PATH` to the host `addons` directory. The private
Compose example mounts it read-only at `/opt/camofox-addons`.

The mounted directory must be an extracted add-on with `manifest.json` at its
root; raw `.xpi` files are not passed to Camofox directly.

### Headless Obsidian Clipping

When the Obsidian Web Clipper add-on is configured, the Camofox wrapper stages a
temporary copy and appends a small bridge to the staged `content.js`. The
private mounted add-on is not modified.

Agents must not use noVNC/VNC for clipping. Use the authenticated Camofox API:

```http
POST /tabs/{tabId}/extensions/obsidian-web-clipper/clip
Authorization: Bearer <CAMOFOX_ACCESS_KEY>
Content-Type: application/json

{
  "userId": "spartan-camofox-main",
  "mode": "saveFile",
  "waitForDownloadMs": 10000,
  "includeData": true
}
```

The route verifies that `tabId` belongs to `userId`, triggers the extension's
existing `saveMarkdownToFile` handler through a one-shot bridge token, and
returns the single new download captured for that tab. Bridge invocation and
download capture run under Camofox's per-tab lock so navigation, close, and
concurrent clip operations cannot race the request.

## noVNC

The Camofox VNC plugin is disabled by default:

```env
CAMOFOX_ENABLE_VNC=0
```

If you need human browser interaction, enable it only in private env. The
private Compose override exposes noVNC through Caddy on
`CAMOFOX_NOVNC_PORT`, with Caddy Basic Auth and the VNC password:

```env
CAMOFOX_ENABLE_VNC=1
CAMOFOX_VNC_PASSWORD=<private-password>
CAMOFOX_NOVNC_PORT=26080
```

Do not expose Camofox's API port `9377` on the host by default.

The default Caddyfile includes the HTTP `:26080` listener used by the private
port mapping. The live-profile helper validates the rendered mapping, internal
noVNC health, and public Caddy reachability. With HTTPS, `TAILSCALE_HOST` must
resolve to `TAILSCALE_IP` and be valid for the configured certificate; there is
no raw-IP fallback.

## Manual Login Profiles

Open a Camofox profile for human login:

```sh
sg-camofox-profile-live main https://accounts.google.com
```

Log in through the printed noVNC URL, then select that profile for Hermes:

```sh
sg-camofox-profile-save main
sg-camofox-profile-use main
sg-browser-mode-apply
```

The printed noVNC URL opens `vnc_lite.html` directly and includes the VNC
password from `CAMOFOX_VNC_PASSWORD` in the URL fragment. The fragment is not
sent to Caddy or websockify. If Camofox is already running and noVNC is healthy,
the helper reuses it instead of rebuilding or recreating the service.

Hermes receives the runtime `CAMOFOX_ACCESS_KEY` needed to call Camofox. It
also receives `CAMOFOX_USER_ID`, `CAMOFOX_SESSION_KEY`, and
`CAMOFOX_ADOPT_EXISTING_TAB` from the selected private env so browser tools use
the same saved Camofox profile instead of a new ephemeral `hermes_*` session.
It does not receive the human account password, 2FA material, or personal API
keys. Use a different profile name, such as `alt`, for isolated cookies and
login state.

## Smoke Tests

Healthcheck:

```sh
sg-camofox-health
```

Protected-route browser smoke:

```sh
sg-camofox-smoke
```

The smoke helper creates a tab through Camofox with bearer auth, navigates to
`https://example.com`, lists tabs, and closes the test session. It is a better
integration check than `/health` because it exercises auth and the browser tab
lifecycle.

## Known Limits

- Camofox does not replace `BROWSER_CDP_URL` or `BROWSER_CDP_MAIN_URL`.
- Hermes `browser_cdp` functionality remains Browserless/Chromium-backed.
- Google Meet realtime audio remains Browserless CDP or explicit host Chrome
  CDP only; Camofox supports transcribe mode.
- Browserless profile seeding stays Browserless-only.
- Camofox is Firefox/Camoufox-based, so site behavior may differ from Chromium.
